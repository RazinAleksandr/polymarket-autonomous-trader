# Post-Mortem

<!-- CLAUDE-EDITABLE: Claude may add examples, refine criteria, annotate decisions -->

## When to Load
Load this skill during Phase E (Learn and Evolve) when analyzing resolved positions and extracting lessons.

## Framework

### Step 1: Gather Resolved Position Data
- Run resolution-parser framework first (load resolution-parser.md if not already loaded)
- For each resolved position, collect: market_id, question, category, entry_price, estimated_prob, confidence, edge, side, outcome, P&L, reasoning at time of entry

### Step 2: Classify the Outcome
For each resolved position, classify into one of:
- **Correct + Profitable**: Predicted right direction AND made money. Validate: was the edge real or lucky?
- **Correct + Unprofitable**: Predicted right direction but lost money (e.g., fees ate the edge, price moved against before resolution). Investigate: was sizing too aggressive? Was entry timing poor?
- **Incorrect + Small Loss**: Wrong direction but position was small. Review: what information was missing or misweighted?
- **Incorrect + Large Loss**: Wrong direction AND significant loss. Deep investigation required: was there overconfidence? Was research shallow?

### Step 3: Root Cause Analysis
For each loss (or suboptimal win):
- **Information gap**: Did we miss a key data point that was available? (e.g., breaking news, insider signal)
- **Reasoning error**: Was the Bull/Bear analysis flawed? (e.g., base rate ignored, recency bias)
- **Sizing error**: Was the position too large for the confidence level?
- **Category mismatch**: Does this category systematically produce worse results?
- **Timing error**: Was the entry too early/late relative to resolution?

### Step 4: Extract Actionable Lessons
Each lesson must be:
- **Specific**: "Crypto markets with < 48h to resolution tend to gap" not "Be careful with crypto"
- **Evidence-backed**: Cite the specific trade (market ID, date, P&L)
- **Actionable**: Must change a future behavior (sizing rule, category filter, research step)

### Step 5: Determine if Golden Rule Warranted
A golden rule is warranted when:
- Same mistake pattern has occurred 2+ times
- Single mistake caused >2% bankroll loss
- The lesson is generalizable across categories
Format: "RULE: [rule text] — Learned from: [trade citation with date and P&L]"

# Calibration Check

<!-- CLAUDE-EDITABLE: Claude may add examples, refine criteria, annotate decisions -->

## When to Load
Load this skill during Phase E (Learn and Evolve) when updating calibration data after resolutions, and during Phase C (Analyze Markets) to check for known calibration biases before estimating probabilities.

## Framework

### Step 1: Load Current Calibration Data
- Read knowledge/calibration.json for historical accuracy data
- Structure: { "overall": { "total_predictions": N, "brier_score": X }, "by_category": { "crypto": { ... }, ... }, "by_confidence_bucket": { "0.5-0.6": { ... }, ... } }

### Step 2: Record New Outcomes (after resolution)
For each newly resolved position:
- Append to calibration.json:
  - category, estimated_prob, actual_outcome (1.0 or 0.0), brier_score, confidence_at_entry, date
- Update running averages:
  - Overall Brier score: sum(brier_scores) / N
  - Category Brier score: same, filtered by category
  - Confidence bucket accuracy: group by stated confidence ranges

### Step 3: Analyze Calibration Quality
- Perfect calibration: when you say 70%, it happens 70% of the time
- Overconfidence: predicted prob consistently higher than actual rate → reduce estimates
- Underconfidence: predicted prob consistently lower than actual rate → increase estimates
- Category bias: some categories systematically miscalibrated
- Calculate: mean_error_pp = mean(|estimated_prob - actual_outcome|) * 100 for each bucket

### Step 4: Generate Calibration Corrections
If a pattern is clear (5+ data points in a bucket):
- Overconfident in crypto by 15pp → "When analyzing crypto markets, subtract 0.15 from initial estimate"
- Underconfident in politics by 8pp → "When analyzing politics markets, add 0.08 to initial estimate"
- Write corrections to calibration.json "corrections" field

### Step 5: Pre-Analysis Calibration Adjustment
When loading this skill during Phase C (before analysis):
- Check if current market's category has a calibration correction
- Apply correction to final estimated_probability after synthesis
- Log: "Applied calibration correction: [category] [direction] [amount]pp"

## Calibration Health Thresholds
| Metric | Healthy | Warning | Unhealthy |
|--------|---------|---------|-----------|
| Overall Brier | < 0.20 | 0.20-0.30 | > 0.30 |
| Category error | < 15pp | 15-25pp | > 25pp |
| N predictions | > 10 | 5-10 | < 5 (too few to judge) |

# Polymarket Autonomous Trader

<!-- OPERATOR-SET: Do not modify this section -->
Paper trading is the default mode. NEVER switch to live trading without explicit user request.

## Quick Reference
- **Venv:** `source .venv/bin/activate` (required before any Python command)
- **Tools dir:** `tools/` — CLI scripts for market discovery, pricing, trading, portfolio
- **State dir:** `state/` — `strategy.md`, `core-principles.md`, `cycles/`, `reports/`

- **Config:** `lib/config.py` loads from `.env` via python-dotenv

## Session Start

Every trading session begins with:

1. Generate cycle ID: `YYYYMMDD-HHMMSS` format using current UTC time
2. Create directories: `mkdir -p state/cycles/{cycle_id} state/reports`
3. Read these files for context:
   - `state/strategy.md` — your evolving trading strategy
   - `state/core-principles.md` — immutable guardrails (NEVER modify)
   - 3 most recent `state/reports/cycle-*.md` files (sorted descending)
   - `knowledge/golden-rules.md` — accumulated trading wisdom (when it exists)
4. Log: "Starting trading cycle {cycle_id}"

## Trading Cycle

### Phase A: Check Positions
<!-- OPERATOR-SET -->
Review all open positions for sell signals, resolved markets, and thesis changes.

1. Run: `python tools/get_portfolio.py --include-risk --pretty`
2. Run: `python tools/check_resolved.py --pretty`
3. For resolved markets: load `.claude/skills/resolution-parser.md` and follow its framework
4. For positions needing sell evaluation: assess current thesis vs entry thesis
5. Execute sells via `python tools/sell_position.py` for positions with invalidated thesis or hit stop-loss

### Phase B: Find Opportunities
<!-- OPERATOR-SET -->
Discover and filter tradeable markets from Polymarket.

1. Run: `python tools/discover_markets.py --limit 20 --pretty`
2. Apply sweet-spot filter: YES price between 0.15-0.85
3. Remove markets where we already hold positions (from Phase A portfolio)
4. Rank by volume_24h descending, take top 5-8 candidates
5. If fewer than 3 pass, relax price range to 0.10-0.90

### Phase C: Analyze Markets
<!-- OPERATOR-SET -->
Deep-dive each candidate market with web research and probability estimation.

1. Load `.claude/skills/evaluate-edge.md` and follow its framework for each candidate

6. Record results to `state/cycles/{cycle_id}/execution_results.json`

### Phase E: Learn and Evolve
<!-- OPERATOR-SET -->
Analyze outcomes, update calibration, write cycle report, evolve strategy.

1. Load `.claude/skills/post-mortem.md` for any resolved positions from Phase A
2. Load `.claude/skills/calibration-check.md` to record outcomes and update calibration data
3. Load `.claude/skills/cycle-review.md` to write the cycle report and update strategy
4. Write cycle report to `state/reports/cycle-{cycle_id}.md`

6. Log cycle summary: markets scanned, analyzed, traded, P&L impact
