# Evaluate Edge

<!-- CLAUDE-EDITABLE: Claude may add examples, refine criteria, annotate decisions -->

## When to Load
Load this skill during Phase C (Analyze Markets) of the trading cycle, before evaluating any market.

## Framework

### Step 1: Market Triage (2 minutes per market)
- Check price sweet spot: YES price between 0.15-0.85 (tradeable range)
- Check volume: minimum $10,000 24h volume for liquidity
- Check end date: must resolve within MAX_RESOLUTION_DAYS (config)
- Check existing positions: skip markets where we already hold a position
- Decision: PASS (too extreme/illiquid/duplicate) or CONTINUE

### Step 2: Bull Case Research (web search required)
- Search for evidence supporting YES outcome
- Search queries: [event name], [key entities + outcome], [recent news + topic]
- Record: 3-5 specific evidence points with sources and dates
- Estimate: Bull-only probability (what would you say if only Bull evidence existed?)
- Weight recent information (last 7 days) more heavily than older data

### Step 3: Bear Case Research (web search required)
- Search for evidence AGAINST YES outcome
- Search queries: [counter-arguments], [historical base rates for similar events], [risks/obstacles]
- Record: 3-5 specific counter-evidence points with sources and dates
- Estimate: Bear-only probability
- Explicitly look for base rates — "How often does [event type] actually happen?"

### Step 4: Synthesis
- Weigh Bull vs Bear evidence by: recency, source credibility, specificity, base rate alignment
- Calculate estimated_probability (0.0-1.0)
- Calculate edge = estimated_probability - market_price (for YES side) or (1 - estimated_probability) - (1 - market_price) (for NO side)
- Calculate confidence (0.0-1.0): high if evidence is strong and concordant, low if ambiguous or sparse
- Decision criteria:
  - |edge| >= MIN_EDGE_THRESHOLD (0.04 per config) → TRADE CANDIDATE
  - |edge| < MIN_EDGE_THRESHOLD → PASS (not enough edge)
  - confidence < 0.3 → PASS regardless of edge (too uncertain)

### Step 5: Recommended Side
- If estimated_probability > market_price → recommend YES
- If estimated_probability < market_price → recommend NO
- Record reasoning: 2-3 sentences summarizing WHY you disagree with the market

## Decision Criteria Summary
| Condition | Action |
|-----------|--------|
| Price outside 0.15-0.85 | PASS — no edge in extreme prices |
| Volume < $10k/24h | PASS — insufficient liquidity |
| Already hold position | PASS — avoid doubling down |
| |edge| < 0.04 | PASS — insufficient edge |
| confidence < 0.3 | PASS — too uncertain |
| All checks pass | TRADE CANDIDATE → proceed to size-position |

## Examples

# Size Position

<!-- CLAUDE-EDITABLE: Claude may add examples, refine criteria, annotate decisions -->

## When to Load
Load this skill during Phase D (Size and Execute) of the trading cycle, after evaluate-edge produces trade candidates.

## Framework

### Step 1: Validate Trade Candidate
- Confirm edge >= MIN_EDGE_THRESHOLD (0.04)
- Confirm confidence >= 0.3
- Confirm market has not been filtered out by position monitor (WATCH status)

### Step 2: Kelly Criterion Calculation
Formula: kelly_fraction = (edge / odds) where:
- edge = estimated_probability - market_price (for YES side)
- odds = market_price / (1 - market_price) for YES, or (1 - market_price) / market_price for NO
- Full Kelly: f* = (p * b - q) / b where p = estimated_prob, q = 1-p, b = payout odds
- Adjusted Kelly: f_adj = f* * KELLY_FRACTION (default 0.25 — quarter-Kelly for conservative sizing)

### Step 3: Confidence Weighting
- Multiply Kelly size by confidence factor: final_fraction = f_adj * confidence
- This naturally reduces position size when analysis is uncertain
- confidence < 0.5 → very small position (exploratory)
- confidence 0.5-0.7 → moderate position
- confidence > 0.7 → standard Kelly-adjusted position

### Step 4: Apply Position Limits
- Maximum single position: 5% of bankroll (from core-principles.md)
- Maximum total exposure: 30% of bankroll across all open positions
- Check remaining capacity: run `python tools/get_portfolio.py --include-risk` to get current exposure
- If remaining_capacity < proposed_size → reduce to remaining_capacity
- Minimum trade size: $5 (CLOB minimum notional)

### Step 5: Calculate Order Parameters
- size_usdc = bankroll * final_fraction (capped by limits above)
- shares = size_usdc / market_price
- Round shares to 2 decimal places
- Verify: size_usdc >= $5 (minimum), size_usdc <= bankroll * 0.05 (maximum)

### Step 6: Risk Assessment
- Calculate max loss: size_usdc (total loss if wrong)
- Calculate potential profit: shares - size_usdc (profit if correct)
- Calculate expected value: estimated_prob * profit - (1 - estimated_prob) * size_usdc
- If EV < 0 → DO NOT TRADE (something is wrong with the math)

## Formulas Reference
| Formula | Expression |
|---------|-----------|
| Edge (YES) | estimated_prob - yes_price |
| Edge (NO) | (1 - estimated_prob) - no_price |
| Full Kelly | (p * b - q) / b |
| Quarter Kelly | full_kelly * 0.25 |
| Confidence-adjusted | quarter_kelly * confidence |
| Position cap | min(kelly_size, bankroll * 0.05, remaining_capacity) |

## Examples

- Quarter Kelly: full * 0.25
- Confidence-adjusted: quarter * 0.85
- Bankroll: $10,000. Position: $300 (3% of bankroll)
- Within 5% cap ($500), within 30% total exposure

# Resolution Parser

<!-- CLAUDE-EDITABLE: Claude may add examples, refine criteria, annotate decisions -->

## When to Load
Load this skill during Phase A (Check Positions) when checking for resolved markets, and during Phase E (Learn and Evolve) when analyzing outcomes.

## Framework

### Step 1: Check for Resolved Markets
- Run: `python tools/check_resolved.py --pretty`
- This queries Gamma API for market.closed status on all open positions
- Output: list of resolved markets with outcome (YES/NO) and resolution details

### Step 2: Record Outcomes
For each resolved market:
- Match to original trade: find the BUY entry in trading.db with market_id
- Extract: entry_price, estimated_prob, edge, confidence, reasoning, category
- Calculate actual outcome: 1.0 if our side won, 0.0 if our side lost
- Calculate P&L: (payout - cost_basis) accounting for fees

### Step 3: Calculate Prediction Accuracy
- Brier score: (estimated_prob - actual_outcome)^2
  - Perfect prediction: 0.0
  - Worst prediction: 1.0
  - Random (0.5 on everything): 0.25
- Error in percentage points: |estimated_prob - actual_outcome| * 100
- Was our side correct: did we predict the right direction?

### Step 4: Categorize the Outcome
- Category: from the original trade (crypto, politics, sports, etc.)
- Edge quality: was the edge real? (correct direction + profitable)
- Overconfidence check: did we overestimate probability? (estimated > actual for winning side)
- Underconfidence check: did we underestimate? (estimated < actual for winning side)

### Step 5: Extract Lessons
For each resolved position, answer:
- What did we get right? (thesis confirmed, timing correct, etc.)
- What did we get wrong? (thesis invalid, timing off, missed information)
- What would we do differently? (different sizing, different entry, pass entirely)
- Should this inform a golden rule? (pattern of similar mistakes)

## Resolution Mechanics by Category
| Category | Typical Resolution | Common Pitfalls |
|----------|-------------------|-----------------|
| Crypto | Price feed at specific date/time | Flash crashes, exchange differences |
| Politics | Official result/announcement | Delayed results, contested outcomes |
| Sports | Final score/result | Overtime, disqualifications |
| Commodities | Price feed at specific date/time | Contract month confusion |
| Entertainment | Award ceremony/official announcement | Ties, rule changes |
| Finance | Official report/data release | Revisions to initial data |
